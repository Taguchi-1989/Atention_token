## 3. UI 設計要因の分析

### 3.1 スコアに影響する UI 要因（重要度順）

| # | UI 要因 | 影響度 | 検出方法 | 人間への影響 |
|---|---------|--------|----------|------------|
| 1 | **要素に id/label がない** | 致命的 | エージェント操作不可 → 失敗 | アクセシビリティ低下、スクリーンリーダー使用不可 |
| 2 | **フィールド数の多さ** | 大 | ステップ数に直結 (v1: 11 vs v2: 2) | 入力の手間、離脱率上昇 |
| 3 | **デフォルト値の有無** | 大 | 入力スキップ可否 | 「何を入れればいいかわからない」の解消 |
| 4 | **視覚的ノイズ（バナー、サイドバー）** | 中 | 入力トークン増加 (12,382 vs 823) | 注意の分散、目的のフィールドが見つけにくい |
| 5 | **冗長フィールド（確認メール等）** | 中 | 不要なステップの発生 | タイプミスの恐怖、手間の増加 |
| 6 | **2段階操作（Confirm→Submit）** | 中 | リトライの発生 (v1 Step 9) | 「どっちを押せばいいの？」の混乱 |
| 7 | **CAPTCHA** | 小〜中 | エージェントは突破不可、人間も手間 | UX の中断、離脱の原因 |
| 8 | **ボタンの視認性** | 中 | 複数ボタン並列 → 迷い | 「Confirm? Submit? Reset?」の判断コスト |

### 3.2 要因別スコア影響の推定

```
inquiry_form のスコア差分析 (v1: 7,741 → v2: 618、差: 7,123)

フィールド数 (18→4)           ████████████████████████████████████████  ~4,000 (56%)
デフォルト値なし→あり           ██████████████████                        ~1,500 (21%)
サイドバー/ノイズ               █████████████                             ~1,000 (14%)
2段階操作 (Confirm→Submit)     ███████                                      ~500 (7%)
その他 (CAPTCHA, 冗長フィールド) ██                                          ~123 (2%)
```

### 3.3 エージェント vs 人間の比較

| 評価観点 | エージェントで検出可能か | 人間の方が得意か | 備考 |
|----------|----------------------|----------------|------|
| フィールド数の多さ | **検出可能** — ステップ数に直結 | 同等 | 両者とも多い = 大変 |
| デフォルト値の効果 | **検出可能** — 入力スキップで即反映 | 同等 | 人間も「埋まってるなら送信」 |
| ボタンの id/label 欠如 | **検出可能** — 操作失敗として顕在化 | 人間は視覚で補完可能 | アクセシビリティの問題 |
| 視覚的ノイズ | 間接的（トークン量） | **人間の方が敏感** | 人間は「うるさい」と感じるがエージェントは処理するだけ |
| 色・コントラスト | **検出不可**（テキストベース） | **人間の方が敏感** | ビジョンモデルなら可能 |
| フォントサイズ・余白 | **検出不可**（テキストベース） | **人間の方が敏感** | CSS はレンダーされない |
| 読み込み速度 | **検出不可**（シミュレータ） | **人間の方が敏感** | 実ブラウザテストなら可能 |
| CAPTCHA の負担 | 操作不可として検出 | **人間の方がストレスを感じる** | 両者にとってネガティブ |
| ボタン3つ並列の混乱 | リトライとして検出 | **人間の方が混乱** | 「Reset」を間違えて押すリスク |

## 4. ベンチマーク指標の定義

### 4.1 認知負荷スコア（Cognitive Load Score）

```
CLS = output_tokens + step_count × 100 + retry_count × 500
```

**解釈:**
- **0〜1,000**: 非常に良い UI — エージェントが迷わず完了
- **1,000〜3,000**: 良い UI — 少数のステップで完了
- **3,000〜10,000**: 改善の余地あり — 複数の判断が必要
- **10,000〜**: 問題のある UI — ループやリトライが頻発

### 4.2 A/B 改善率

```
改善率 = (v1_score - v2_score) / v1_score × 100%
```

| ベンチマーク | v1 | v2 | 改善率 | 判定 |
|------------|-----|-----|--------|------|
| expense | 2,222 | 628 | 72% | 大幅改善 |
| inquiry | 7,741 | 618 | 92% | 劇的改善 |
| cart | 2,882 | 1,094 | 62% | 大幅改善 + 成功率向上 |

### 4.3 操作効率指標

| 指標 | 計算 | 意味 |
|------|------|------|
| ステップ効率 | v2_steps / v1_steps | 低いほど v2 が効率的 |
| トークン効率 | v2_tokens / v1_tokens | 低いほど v2 が明快 |
| 成功率差 | v2_success - v1_success | 正なら v2 が安定 |

### 4.4 UI 品質チェックリスト（エージェント検出可能な項目）

- [ ] 全 input/select/button に一意の id が付与されているか
- [ ] フィールド数は必要最低限か（不要なフィールドがないか）
- [ ] デフォルト値・プレースホルダが適切に設定されているか
- [ ] 送信ボタンは1つだけか（Confirm/Submit/Reset の並列を避ける）
- [ ] CAPTCHA は本当に必要か
- [ ] 確認メール等の冗長フィールドは排除されているか
- [ ] サイドバー等のノイズ要素は最小限か
- [ ] ドロップダウンの選択肢数は適切か
- [ ] 成功/エラーメッセージが明確に表示されるか

## 5. 結論

### 5.1 Attention Ledger は UI 品質差を定量的に検出できる

3つの A/B ペア全てで、**良い UI (v2) のスコアが悪い UI (v1) より 62〜92% 低い**。この差は統計的に明確であり、UI 設計の品質差が認知負荷スコアに反映されることを実証した。

### 5.2 最も影響の大きい UI 要因

1. **フィールド数の削減** — スコアに最大の影響（inquiry: 18→4 で 92% 改善）
2. **適切なデフォルト値** — 入力ステップをゼロにできる
3. **要素の id/label 付与** — エージェント操作の成否を決定（cart v1 失敗の根本原因）

### 5.3 エージェント評価の限界と補完

エージェントは **操作の効率性** と **要素の識別可能性** を正確に評価できるが、**視覚的な美しさ** や **感情的な快適さ** は評価できない。

**推奨: エージェント評価 + 人間の SUS アンケートの組み合わせ**

```
総合 UI 品質スコア = α × エージェント認知負荷スコア + β × SUS スコア(反転)
```

エージェント評価は自動・反復・客観的。人間評価は主観的だが感覚的な品質を捉える。両方を組み合わせることで、UI 品質の全体像が見える。
