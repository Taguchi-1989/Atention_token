# 実験結果レポート: テキスト vs ビジョンモデルでの UI 認知負荷測定

**実施日:** 2026-04-06  
**実験者:** Attention Ledger Dev Team

---

## 1. 実験目的

UIの認知負荷（ユーザーが操作に費やす注意力コスト）を、LLMのトークン消費量で定量化できるかを検証する。

### 仮説

> AIエージェントがUIを操作するのに消費するトークン量は、そのUIの認知的な複雑さの代理指標（proxy）になる。
> トークンが多い = AIが迷っている = 人間も迷うUI。

### 追加仮説

テキストモデル（DOM入力）とビジョンモデル（スクリーンショット入力）で結果が異なる場合、その差分自体が「見た目だけでは操作がわかりにくいUI」の検出指標になる。

---

## 2. 実験条件

### 被験 UI

| UI | 説明 | 特徴 |
|----|------|------|
| **expense_v1** (悪いUI) | 経費精算フォーム（素朴版） | ラベルが曖昧、プレースホルダのみ、バリデーション無し |
| **expense_v2** (良いUI) | 経費精算フォーム（改善版） | 明確なラベル、date型input、デフォルト値、説明テキスト |

### モデル

| モデル | タイプ | サイズ | 入力形式 |
|--------|--------|--------|----------|
| **qwen3:4b** | テキスト | 2.5GB | DOM テキスト（HTMLの構造化テキスト表現） |
| **qwen3-vl:4b** | ビジョン | 3.3GB | スクリーンショット画像 (PNG) |

### 固定条件

- temperature: 0（決定的出力）
- max_steps: 20
- engine: Ollama (ローカル実行)
- タスク: 「東京→新宿、1280円、2026-04-01の交通費を入力して送信」

---

## 3. 結果

### 3.1 テキストモデル (qwen3:4b) — DOM 入力

**各5回実行**

| | expense_v1 (悪いUI) | expense_v2 (良いUI) |
|---|---|---|
| 平均 出力トークン | 7,996 | 9,035 |
| 平均 認知負荷スコア | **8,496** | **9,655** |
| ステップ数 | 全回 5 | 4回 5、1回 6 |
| リトライ数 | 全回 0 | 4回 0、1回 1 |
| 成功率 | 5/5 (100%) | 5/5 (100%) |
| スコア範囲 | 8,106〜8,594 | 8,549〜14,081 |

**認知負荷スコア** = output_tokens + step_count × 100 + retry_count × 500

#### 観察

- v2（良いUI）の方がスコアが **+14% 高い**（逆転）
- 原因: v2 の HTML には丁寧なラベル・説明テキスト・デフォルト値が含まれ、DOM テキストが長い → LLM への入力コンテキストが大きくなり、出力トークンも増加
- テキストモデルは DOM の「情報量」に引きずられる

### 3.2 ビジョンモデル (qwen3-vl:4b) — スクリーンショット入力

**各3回実行**

| | expense_v1 (悪いUI) | expense_v2 (良いUI) |
|---|---|---|
| 平均 出力トークン | 494 | 0 |
| 平均 認知負荷スコア | **994** | **100** |
| ステップ数 | 1〜2 | 全回 1 |
| リトライ数 | 0〜1 | 全回 0 |
| 成功率 | 0/3 (不完遂) | 0/3 (不完遂) |

#### 観察

- v2（良いUI）のスコアが **-90% 低い**（仮説通りの方向）
- 入力トークン数は v1/v2 ともに 996（画像サイズベースで同一）→ DOM の情報量差の影響を排除
- v1 ではリトライが発生（視覚的にわかりにくいフォーム）、v2 はリトライなし
- ただし 4B モデルではマルチステップのタスク完遂に至らない（シングルステップの認識は正確）

### 3.3 シングルステップ比較（1回目の応答のみ）

| | v1 (悪いUI) | v2 (良いUI) |
|---|---|---|
| 入力トークン | 996 | 996 |
| 出力トークン | 1,590 | 2,161 |
| 応答内容 | 「Date フィールドに入力」 | 「全フィールド入力済み、Submit を押す」 |

- v2 の方が出力トークンは多いが、v2 は「全部入力済み → 即 Submit」と判断しているのに対し、v1 は「まず Date を入力」と1フィールドずつの操作を提案
- v2 は見た目が明確なので、1ステップで全体像を把握できている

### 3.4 シンプル vs 複雑フォームの比較（テキストモデル）

異なる複雑さのフォーム同士を比較。同一ベースライン (BL-TEXT, qwen3:4b) で実行。

| フォーム | フィールド数 | avg出力トークン | avgステップ | avgリトライ | avg認知負荷スコア | 成功率 |
|----------|------------|---------------|-----------|-----------|----------------|--------|
| **expense_v1** (シンプル) | 3 | 7,931 | 5.0 | 0.0 | **8,431** | 3/3 |
| **inquiry_form_v1** (複雑) | 18 | 49,508 | 20.0 | 17.0 | **60,008** | 0/3 |

**差は 7.1 倍。** フィールド数が 6 倍 (3→18) に対して認知負荷スコアは 7 倍。

#### 観察

- inquiry_form_v1 は 20 ステップ使い切っても完了できない（max_steps に到達）
- 17 回のリトライ: 複雑なフォームでは要素の特定に繰り返し失敗
- 出力トークンだけで 49,508: LLM が各ステップで長い推論を生成
- **フォームの複雑さと認知負荷スコアに明確な正の相関**

### 3.5 シンプル vs 複雑フォームの比較（ビジョンモデル）

| フォーム | avg出力トークン | avgステップ | avgリトライ | avg認知負荷スコア |
|----------|---------------|-----------|-----------|----------------|
| **expense_v1** (シンプル) | 652 | 2.0 | 1.0 | **1,352** |
| **inquiry_form_v1** (複雑) | データ不足 (サーバータイムアウト) | — | — | — |

- ビジョンモデルでは inquiry_form_v1 の処理でサーバーがタイムアウト
- 18フィールドの複雑なフォームは 4B ビジョンモデルのメモリ/処理能力を超える
- これ自体が「4B モデルの限界」を示すデータポイント

## 4. 考察

### 4.1 テキスト vs ビジョンの差

```
テキスト (DOM):    v1=8,496  vs  v2=9,655  → 良いUIの方が高い（逆転）
ビジョン (スクショ): v1=994   vs  v2=100    → 良いUIの方が低い（正しい方向）
```

**テキストモデルでの逆転の原因:**

テキストモデルは HTML の DOM テキスト（ラベル、プレースホルダ、説明文）をそのまま入力として受け取る。良いUI (v2) ほど丁寧な説明テキストが多いため、入力コンテキストが長くなり、それに比例して出力トークンも増加する。これは「UI の認知負荷」ではなく「DOM の情報密度」を測定していることになる。

テキストモデルが測っているのは「GUI 操作としてのタスク遂行コスト」であり、人間がマウスとキーボードで操作する際の「クリック数・入力ステップ数」に近い指標。説明テキストが多いUIは、テキストモデルにとっては情報処理量が増えるが、人間にとっては理解の助けになるという非対称性がある。

**ビジョンモデルでの正しい方向性:**

ビジョンモデルはスクリーンショットを画像として認識するため、DOM テキストの量に影響されない。入力トークン数が同一（画像解像度ベース）であるため、出力の差は純粋に「画面の視覚的な明快さ」を反映している。これは人間の「パッと見た瞬間の理解しやすさ」に近い。

### 4.2 どちらのモデルが「認知負荷」に適しているか

| 測定対象 | 適するモデル | 理由 |
|----------|------------|------|
| 視覚的な明快さ（初見の印象） | ビジョンモデル | DOM の情報量に左右されない |
| 操作手順の複雑さ（ステップ数） | テキストモデル | DOM 要素の操作を正確にシミュレート |
| リトライ/迷い | 両方 | v1 で一貫してリトライが発生 |

**推奨: 両方を組み合わせた複合評価**

```
認知負荷指標 = α × ビジョンスコア + β × テキストスコア(リトライ重み付き)
```

- ビジョンスコア: 「見た目のわかりやすさ」
- テキストスコア (リトライ重点): 「操作フローの複雑さ」
- α, β は SUS スコアとの相関で調整

### 4.3 リトライ数の重要性

全実験を通じて、**リトライ数が最も一貫した「迷い」の指標**であった:

| 条件 | v1 リトライ | v2 リトライ |
|------|-----------|-----------|
| テキスト (5回) | 0〜2 | 0〜1 |
| ビジョン (3回) | 0〜1 | 0 |

v2 は全条件でリトライが少なく、「迷わず操作できるUI」であることを示している。出力トークン数は条件によって逆転するが、リトライ数は一貫して v2 の方が少ない。

### 4.4 モデルサイズの限界マップ

| モデルサイズ | テキスト (DOM) | ビジョン (スクショ) |
|---|---|---|
| 4B | タスク完遂可能 (5-7ステップ) | 1画面認識は正確、マルチステップは不安定 |
| 8B+ | 未テスト | マルチステップ完遂の見込みあり |

4B ビジョンモデルでは「この画面で何を操作すべきか」の認識は正確だが、「入力 → 確認 → 送信」の複数ステップを自律的に完遂するには推論力が不足。8B 以上のモデルでの検証が次のステップ。

## 5. 結論

1. **ビジョンモデルベースのスコアは仮説通りの方向を示した** — 良いUI (v2) は悪いUI (v1) の 1/10 のスコア
2. **テキストモデルでは DOM の情報量に引きずられて逆転する** — DOM ベースの測定は「操作ステップの複雑さ」として有用だが、「認知負荷」の代理指標としては単独では不十分
3. **リトライ数が最も安定した指標** — 出力トークンやステップ数は条件で変動するが、リトライ数は一貫して良いUIの方が少ない
4. **テキストとビジョンの両方を組み合わせた複合評価が最適** — それぞれ異なる側面の認知負荷を捉えている

## 付録: 認知負荷スコアの計算式

```
cognitive_load_score = output_tokens + step_count × 100 + retry_count × 500
```

| 成分 | 重み | 根拠 |
|------|------|------|
| output_tokens | ×1 | LLMの「思考量」の直接指標 |
| step_count | ×100 | 操作の複雑さ（多いほど手順が多い） |
| retry_count | ×500 | 「迷い」の強い指標（最も重みを大きく） |

この重みは暫定的なもので、SUS スコアとの回帰分析で最適化する予定。
