# Attention Ledger Local

## 要件定義書 v0.1（MVP）

---

## 1. 目的 / Vision

本プロジェクトは、
**業務システム・UI・業務フローによって浪費されている「人の注意力」を、AIトークンを代理通貨として定量化し、改善前後の差分を測定可能にする OSS ツール**を構築する。

### コア思想

* 人を評価しない
* 組織を評価しない
* **構造（UI・フロー・設計）だけを評価する**
* 絶対値ではなく **差分で測る**
* 初見の人間の代理として **AIエージェントを使う**

---

## 2. スコープ（MVP）

### 対象

* 自作アプリや業務システムの A/B UI
* 単一タスク（5〜30操作程度）
* 即時性不要、バッチ評価

### 対象外（MVPではやらない）

* リアルタイム監視
* 人事評価用途
* 自動UI操作の完全自律
* クラウド必須設計

---

## 3. 全体構成

```
attention-ledger/
 ├─ core/
 │   ├─ agent/
 │   ├─ llm/
 │   ├─ task/
 │   ├─ execute/
 │   ├─ metrics/
 │   └─ baseline/
 ├─ cli/
 ├─ storage/
 ├─ report/
 └─ docs/
```

---

## 4. 基本アーキテクチャ

### 原則

* **完全ローカル実行**
* **LLM APIは差し替え可能**
* **計測条件はベースラインとして固定**

### 構成

```
[CLI]
  ↓
[Task Scenario]
  ↓
[Initial-View AI Agent]
  ↓
[LLM Adapter]
  ↓
[Execution Engine]
  ↓
[Metrics Collector]
  ↓
[Ledger / Baseline Store]
```

---

## 5. LLMアダプタ要件

### 抽象インターフェース

```python
class LLMAdapter:
    def generate(prompt: str) -> LLMResponse
    def token_usage(response: LLMResponse) -> TokenUsage
```

### MVP対応想定

* Ollama
* llama.cpp

### 固定条件

* model name
* tokenizer
* temperature = 0
* max output tokens
* system prompt hash

※ すべて Baseline に記録すること

---

## 6. 初見AIエージェント仕様（重要）

### 役割

* 初見の人間として振る舞い
* タスク完了を目指す
* 推測せず、迷えば探索する

### 制約

* 出力は **短く・構造化**
* 冗長な説明は禁止
* 判断・探索・失敗をログとして残す

### 出力形式（例）

```json
{
  "action": "click | input | search | retry | done",
  "target": "button_name | field_name",
  "confidence": "low | medium | high",
  "note": "short reason"
}
```

※ この prompt は **評価基準そのもの**であり、
バージョン管理必須。

---

## 7. タスクシナリオ定義

### 形式（YAML）

```yaml
task_id: EXPENSE_INPUT_V1
description: 経費精算の交通費入力
start_condition: logged_in_home
goal_condition: expense_saved
input_data:
  date: 2026-04-01
  amount: 1280
  route: 東京-新宿
```

### 要件

* 再現可能であること
* 入力値固定
* 成功条件が明確

---

## 8. 実行エンジン要件

* タスクを 1 回ずつ実行
* 途中失敗・リトライを許容
* 各ステップで以下を必ず記録

  * LLM呼び出し
  * token usage
  * step number
  * retry flag

---

## 9. メトリクス定義（MVP必須）

### Core Metrics

* total_tokens
* input_tokens
* output_tokens
* step_count
* retry_count

### Optional（後続）

* decision_unit_level (DU-1〜DU-5)
* backtrack_count
* error_event_count

---

## 10. ベースライン管理

### Baseline定義

```yaml
baseline_id: BL-2026-H1
model: llama-7b-q4
engine: ollama
system_prompt_hash: xxxx
temperature: 0
created_at: 2026-04-01
```

### ルール

* ベースラインは immutable
* 半期ごとに新規作成
* 実行結果は必ず baseline_id を持つ

---

## 11. レジャー（台帳）要件

### 保存内容

* task_id
* baseline_id
* executed_at
* metrics
* success / failure

### 制約

* append-only
* SQLite / ローカルDB可
* 後から改ざんできない設計

---

## 12. 人リソース換算（MVP）

### 前提

* 人の実測時間（平均 / P90）は外部入力
* 換算は **後段処理**

### 換算式

```
k = 人時間（分） / total_tokens
削減時間 = k × Δtokens
```

※ MVPでは CSV 手入力で可

---

## 13. CLI要件（最低限）

```bash
ledger baseline create
ledger task run <task_id> --baseline BL-2026-H1
ledger report diff --task EXPENSE_INPUT_V1 --A BL-A --B BL-B
```

---

## 14. 非機能要件

### セキュリティ

* 外部通信なし
* ログ削除可能

### 再現性

* 条件固定
* 同一傾向が出ることを重視

### OSS前提

* Apache-2.0 or MIT
* 設計思想は README に明記

---

## 15. 成功条件（PoC完了ライン）

* 同一タスクで UI A / B のトークン差分が出る
* 人時間差分と方向が一致する
* 「改善効果を事前に推定できる」状態になる
