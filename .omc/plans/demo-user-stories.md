# Attention Ledger デモ計画 & ユーザーストーリー

## 1. ユーザーストーリー（3つの具体的シナリオ）

---

### ストーリー 1: 経費精算フォームの改善検証

**ペルソナ:** 田中さん — 社内システム開発チームのUXデザイナー  
**状況:** 経費精算フォームのリニューアルを提案したが、「使いやすくなった」という定量的な根拠がない。上司に「感覚じゃなくてデータで見せて」と言われた。

**ツールで行うこと:**
1. 現行の経費精算フォーム（expense_v1）をAIエージェントに操作させる
2. 改善案のフォーム（expense_v2）も同じタスクで実行する
3. 実行台帳の「比較」タブで両者のトークン消費量を並べる

**得られるインサイト:**
- v1: フォームにスタイルなし、日付フォーマットの指示が曖昧 → エージェントが迷い、トークン消費が多い
- v2: デフォルト値あり、date型input、明確なラベル → エージェントが迷わず完了、トークン大幅削減
- 「AIが迷う = 人間も迷う」という仮説を数値で裏付けられる

**デモでの見どころ:**
- v1の素朴なHTMLとv2の洗練されたフォームの対比
- トークン数の差がそのまま「認知負荷の差」として可視化される

---

### ストーリー 2: ECサイトの情報過多 vs シンプルUI

**ペルソナ:** 鈴木さん — ECプラットフォームのプロダクトマネージャー  
**状況:** 商品ページの情報量を増やせば売上が上がると思っていたが、カート投入率が低下している。「情報過多」が原因かどうか検証したい。

**ツールで行うこと:**
1. 「Sony WH-1000XM5をカートに追加する」というタスクで、ごちゃごちゃしたECページ（shopping_cart_v1）を実行
2. 同じタスクをシンプルなECページ（shopping_cart_v2）で実行
3. ステップ数・リトライ数・トークン消費を比較

**得られるインサイト:**
- v1: フラッシュセールバナー、サイドバーフィルター、スポンサー広告、小さいフォント → エージェントが目的の商品を探すのに大量のステップとトークンを消費
- v2: クリーンなリスト表示、明確なAdd to Cartボタン → 最小ステップで完了
- 「UIノイズが多いほど認知コストが増える」ことを定量的に証明

**デモでの見どころ:**
- v1のHTMLを実際にブラウザで開くと「よくある悪いECサイト」感が強烈
- エージェントの実行ログを見ると、v1では迷走する過程が可視化される

---

### ストーリー 3: 問い合わせフォームの入力項目数とUX

**ペルソナ:** 佐藤さん — SaaS企業のカスタマーサクセス責任者  
**状況:** 問い合わせフォームの完了率が30%しかない。項目を減らすべきか、現状のままで良いかデータが欲しい。

**ツールで行うこと:**
1. 18項目+CAPTCHA+利用規約チェックの問い合わせフォーム（inquiry_form_v1）を実行
2. 4項目のシンプルフォーム（inquiry_form_v2）を実行
3. 比較チャートでdelta（差分）を確認

**得られるインサイト:**
- v1: 18フィールド、CAPTCHA、確認ボタン→送信の2段階 → エージェントのステップ数が爆発、リトライも発生
- v2: 名前・メール・カテゴリ・メッセージの4項目のみ → 数ステップで完了
- 項目数と認知負荷の相関が一目瞭然

**デモでの見どころ:**
- inquiry_form_v1の項目の多さは見た瞬間に「うわっ」となる
- トークン消費の差が最も劇的になるペア

## 2. デモフロー（ステップ・バイ・ステップ）

### 前提条件
```bash
# ターミナル1: バックエンド起動
cd python
uvicorn attention_ledger.api.main:app --reload --port 8000

# Next.jsは事前にビルド済み（out/ディレクトリがstaticとしてサーブされる）
```
ブラウザで `http://localhost:8000` を開く

---

### Phase 1: 最初のインパクト（2分）

**ステップ 1:** ダッシュボードを確認
- トップページの「AttentionLedger」ヘッダーとステータスが表示される
- 「AIエージェントを使って、ソフトウェアの認知負荷を定量化」のキャッチコピーを確認
- まだ実行履歴がない状態

**ステップ 2:** 「タスク実行」ボタンをクリック → タスクライブラリへ

---

### Phase 2: 悪いUIを体験（3分）

**ステップ 3:** タスクライブラリで以下を確認
- EXPENSE_INPUT_V1, EXPENSE_INPUT_V2 などのタスクカードが並んでいる
- 各カードに「開始条件」と「ゴール」が表示されている

**ステップ 4:** ベースラインを作成
- 右上の「+」ボタンで新規ベースライン作成
- `BL-BAD-UI` という名前で作成

**ステップ 5:** EXPENSE_INPUT_V1 の「分析実行」をクリック
- モックモードがONになっていることを確認（トグルが右側）
- 実行ログモーダルが開き、エージェントの思考過程がリアルタイムで流れる
- 「Step 1... Step 2...」とアクションが表示される
- 完了まで待つ

**ステップ 6:** 続けて SHOPPING_CART_V1, INQUIRY_FORM_V1 も実行

---

### Phase 3: 良いUIを体験（3分）

**ステップ 7:** ベースラインを切り替え
- 新規ベースライン `BL-GOOD-UI` を作成
- 選択する

**ステップ 8:** EXPENSE_INPUT_V2 の「分析実行」をクリック
- 実行ログを観察 → v1より明らかに少ないステップで完了する

**ステップ 9:** SHOPPING_CART_V2, INQUIRY_FORM_V2 も実行

---

### Phase 4: 比較で感動（3分）

**ステップ 10:** 左メニューの「実行台帳」をクリック

**ステップ 11:** 「比較」タブを選択
- ベースライン A に `BL-BAD-UI` を選択
- ベースライン B に `BL-GOOD-UI` を選択
- 棒グラフが表示される: total_tokens, step_count, retry_count の各指標でBの方が圧倒的に少ない
- Delta行で「-XX%」が緑色で表示される → 改善率が一目でわかる

**ステップ 12:** 「トレンド」タブで個別タスクの推移も確認可能
- タスクを選択すると、実行ごとのメトリクス推移が折れ線で見える

**ステップ 13:** CSV出力
- 「全件CSV出力」ボタンで生データをダウンロード可能

---

### Phase 5: まとめ（1分）

**語るべきメッセージ:**
> 「AIエージェントがUIを操作するのに消費するトークン量は、そのUIの認知的な複雑さの代理指標（proxy）になります。
> トークンが多い = AIが迷っている = 人間も迷うUI。
> これにより、主観的な『使いやすさ』を客観的な数値に変換できます。」

## 3. オンボーディング改善アイデア

### ランディングページに伝えるべきこと
現在のトップページはダッシュボード（統計表示）になっている。初回訪問者向けに以下を追加すべき：

1. **ワンライナー説明:** 「UIの使いやすさを、AIの"迷い度"で測る」
2. **3ステップの図解:** タスク定義 → AI実行 → 比較・分析
3. **「今すぐ試す」ボタン:** クリックすると expense_v1 vs v2 のデモが自動で走り、結果画面に遷移

### 「ガイドツアー」機能
- 初回アクセス時にオーバーレイで4ステップのガイドを表示
  1. 「これがタスクカードです。AIエージェントに操作させるシナリオを定義します」
  2. 「ベースラインを選んで実行ボタンを押すと、AIが操作を開始します」
  3. 「実行ログでAIの思考過程をリアルタイムに見られます」
  4. 「実行台帳で結果を比較。トークン数の差がUI品質の差です」

### 「マジック体験」の設計
- **ワンクリックデモ:** ダッシュボードに「デモを実行」ボタンを設置。押すと裏でv1とv2を両方実行し、完了後に比較画面を自動表示
- **HTMLプレビュー:** 実行前に「このUIをAIが操作します」として実際のHTMLをiframeでプレビュー表示。v1の"ヤバさ"を視覚的に体感してから実行
- **スコアカード:** 実行完了後に「認知負荷スコア: 78/100 → 改善後: 32/100」のような分かりやすい一枚絵サマリー

## 4. エンドツーエンドで動かすために今足りないもの

### 4.1 動作確認済みの部分（問題なし）
- [x] YAMLタスク定義 6つ（v1/v2 x 3ペア）がすべて `python/tasks/` にある
- [x] 対応するHTML 6つが同じディレクトリにある
- [x] `ExecutionEngine._load_simulator()` が `task_id.lower() + ".html"` で自動マッチする
- [x] APIの `/tasks/{task_id}/run` エンドポイントが背景実行をサポート
- [x] フロントの「タスクライブラリ」でカード表示・実行・ログ表示ができる
- [x] 「実行台帳」に比較チャートとトレンドチャートがある
- [x] CSV出力機能がある

### 4.2 修正が必要な箇所

#### (A) タスクID大文字/小文字のマッチング問題 [重要]
- YAMLの `task_id` は `EXPENSE_INPUT_V1`（大文字）
- `_load_simulator()` は `task_id.lower() + ".html"` で検索 → `expense_input_v1.html`
- しかしHTMLファイル名は `expense_v1.html`（`_input_` がない）
- **結果: HTMLシミュレータが見つからず、フォールバックの文字列ベース実行になる**
- 修正案: YAMLのtask_idを `expense_v1` に変更するか、HTMLファイル名を `expense_input_v1.html` にリネーム

各ペアの不一致：
| YAML task_id | lower化 | HTMLファイル名 | マッチ |
|---|---|---|---|
| EXPENSE_INPUT_V1 | expense_input_v1 | expense_v1.html | NG |
| EXPENSE_INPUT_V2 | expense_input_v2 | expense_v2.html | NG |
| SHOPPING_CART_V1 | shopping_cart_v1 | shopping_cart_v1.html | OK |
| SHOPPING_CART_V2 | shopping_cart_v2 | shopping_cart_v2.html | OK |
| INQUIRY_FORM_V1 | inquiry_form_v1 | inquiry_form_v1.html | OK |
| INQUIRY_FORM_V2 | inquiry_form_v2 | inquiry_form_v2.html | OK |

→ **経費精算ペアだけ不一致**。YAMLの `task_id` を `EXPENSE_V1` / `EXPENSE_V2` に変更するのが最小限の修正。

#### (B) ベースラインとA/B比較のワークフローが手動 [中程度]
現在の比較機能はベースライン単位。デモシナリオでは「同じタスク（経費入力）のv1 vs v2」を比較したいが、現在の実装は「同じタスクの異なるベースライン」を比較する設計。

つまり、デモフローでは：
- BL-BAD-UI で v1系タスクを実行
- BL-GOOD-UI で v2系タスクを実行
- 比較タブで BL-BAD-UI vs BL-GOOD-UI を選ぶ

→ これは **すでに動く** が、異なるタスクID（EXPENSE_V1 vs EXPENSE_V2）のデータが混在するため、比較の正確性に欠ける。
→ 理想は「タスクID x ベースライン」のクロス集計だが、デモ目的ならベースライン単位で十分。

#### (C) デモ用の初期データがない [軽微]
- 初回起動時はデータベースが空でダッシュボードに何も表示されない
- デモ用にシードデータを入れるスクリプトがあると良い
- もしくは「ワンクリックデモ」ボタンで自動実行

#### (D) HTMLプレビュー機能がない [Nice-to-Have]
- 現在、実行前にどんなUIなのか確認する手段がない
- タスクカードにプレビューボタンを追加し、HTMLをiframeで表示できると良い

#### (E) CORS設定 [デモ環境依存]
- CORSは `ATTENTION_LEDGER_DEV` 環境変数がセットされている場合のみ有効
- Next.jsのstaticが同一オリジン（port 8000）からサーブされる構成なら不要
- `.env.example` に記載があるか確認し、README で明記する

### 4.3 推奨する修正優先順

| 優先度 | 項目 | 工数 | 効果 |
|---|---|---|---|
| P0 | (A) expense YAMLのtask_id修正 | 5分 | HTMLシミュレータが動くようになる |
| P1 | (C) デモ実行用スクリプト or ワンクリックボタン | 1-2時間 | 初回体験が劇的に改善 |
| P2 | (D) HTMLプレビュー機能 | 2-3時間 | v1/v2の差を視覚的に体感できる |
| P3 | (B) タスクID x ベースラインのクロス比較 | 4-6時間 | 分析の正確性向上 |

## 5. デモ全体のナラティブ構成

```
[導入] 「UIの"使いやすさ"を数値化できたら、どうしますか？」

[課題提起] ユーザビリティテストは時間もコストもかかる。
           ヒューリスティック評価は専門家の主観に依存する。
           SUSスコアは事後評価で、設計段階では使えない。

[解決策] Attention Ledger は、AIエージェントにUIを操作させ、
         その「迷い度」（トークン消費量）を認知負荷の代理指標にする。

[デモ]   3つのシナリオで実証
         1. 経費精算フォーム: ラベルとデフォルト値の効果
         2. ECサイト: 情報ノイズが認知コストに与える影響
         3. 問い合わせフォーム: 入力項目数と完了コストの相関

[結果]   すべてのペアでv2（改善版）のトークン消費が大幅に少ない。
         → AIが迷わない UI = 人間にも優しい UI

[締め]   「トークン消費量という新しいUXメトリクスで、
          デザインの品質を設計段階から定量評価できます。」
```
